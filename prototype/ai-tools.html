🧩 AI 工具配置

V5 不再把 AI 能力堆成一串零散按钮，而是把读写、治理、发布和连接器收束成 6 个能力族，再通过预览闸门和异步任务落到真实产品链路。

为什么要重构

旧式扁平工具容易把权限打平，让搜索、分析、写入、发布混成一把大锤。
真实系统需要区分同步读取、异步生成、预览审阅、事件回推，才能让 AI 用户与人类用户平等协作而不越界。
V5 将 AI 能力直接接到版本、变更请求、语言版本和发布链路，而不是独立悬空。

最佳接口形态

同步接口
上下文读取、文档读取、列目录、权限检查、关键词搜索
异步任务
摘要、大纲、FAQ、翻译、SEO 优化、批量 embedding
预览闸门
任何写入、删除、发布前先生成 preview diff，再走治理链审批

1. Context

看当前位置、看权限边界、看当前 Space 与文档树。

context.get / context.navigate / permission.inspect

2. Knowledge

列内容、读内容、做混合检索与相关推荐。

content.list / content.read / search.hybrid / search.related

3. Drafting

创建草稿、块级改写、对话转文档，但先写到草稿层。

draft.create / draft.patch / dialogue.capture / block.transform

4. Governance

快照、Diff、变更请求预览与审批提交流。

version.snapshot / change_request.preview / approval.submit

5. Publishing

SEO 体检、发布预览、多语言发布和站点释放。

seo.inspect / publish.preview / publish.release

6. Connectors

Git Sync、Webhook、外部 Skill、Capability Manifest。

git.sync / connector.invoke / knowledge.retrieve / webhook.emit

典型执行链

对话生成
对话转文档 → 草稿预览 → 版本快照 → 变更请求 → 发布
跨语言治理
读取默认语言 → 翻译工作流 → 术语库校验 → 语言版本发布 → hreflang / SEO 检查
AI 平权协作
AI 用户与人类用户共享角色模板，但执行动作再叠加速率、范围、审批边界

Skill 注册链

注册 AI 账号
Skill 通过 `POST /users` 创建 `user_type=ai` 的系统用户，再签发 service credential。
Manifest 暴露
通过 `/.well-known/capabilities.json` 与 `/.well-known/skill-manifest.json` 暴露能力边界、审批要求与异步任务模型。
不额外造角色
Skill 只是接入协议层，真正执行者仍是 AI 用户，与人类共享角色和权限。

Skill 管站链

读站点
先读 Site / Section / Linked Space，定位当前发布结构和公开入口。
做预览
对站点级调整先生成预览，必要时创建发布计划和 GitHub 同步任务。
走审批
发布、合并、恢复仍走治理链，Skill 不能直接绕开 Change Request 和 Release 记录。

安全原则

不给 AI 裸露 delete / publish 这类高风险动作，写入要先落草稿或预览。
AI 用户和人类用户共享角色体系，但自动执行还要叠加速率、范围和审批边界。
结果回执进入通知中心、AI 任务中心和变更请求，不让执行结果消失在对话框里。
