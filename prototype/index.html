知识空间

管理个人 / 组织 Space，明确公共 / 私有策略、成员范围和文档边界。

模板中心

管理 PRD、API、FAQ、帮助中心等模板，支持制作、切换、升级与回滚。

文档中心

在集合与 Space 中管理文档树，支持从空白、模板或对话快速起稿。

AI 任务中心

承载摘要、翻译、SEO 检查、对话转文档等异步任务与审计回执。

语言版本

用 `doc_group` 管多语言，不复制整套 Space；支持回退语言和跨语言检索。

发布 & SEO

管理发布站点、站点分区、公开阅读页、SEO 片段和发布目标。

核心操作路径

1
先选 Scope
个人工作区适合沉淀与试写，组织适合协作、审批、发布与 GitHub 同步。
2
先建 Space，必要时放入 Collection
Space 是真正的内容主容器；Collection 只是 Scope 下的可选管理分组。
3
用模板或对话起稿
模板、空白文档、对话转文档都进入同一编辑链，而不是平行入口。
4
进入版本与审批
版本、CR、AI 任务、翻译复审和回收站进入统一治理链。
5
再发布到 Site
Site 不拥有内容，只链接公共 Space，并通过站点分区输出公开阅读体验。

内容结构（内部）

Scope
个人工作区或组织，是所有权、成员、模板和审批策略的边界。
Collection
内容集合，用来组织多个 Space，本身不直接承载页面正文或发布导航。
Space
真正的内容容器，承载文档树、版本、CR、标签、文件、权限和 AI 任务。
Document
支持目录树、块级编辑、多语言文档组、回收站和恢复。

发布结构（外部）

Site
发布站点负责域名、品牌、SEO、导航和公开访问入口。
Section
站点分区，用于把已发布的 Space 按产品文档、帮助中心、开放平台组织起来。
Linked Space
发布时把公共 Space 链接到 Site，而不是把 Site 放到内容链最上层。
Public Reader
公开用户看到的是 Site 下的已发布 Space 和文档，不会进入内部协作态。

单人如何开始

先在个人工作区建立 2 到 3 个 Space：研究沉淀、写作草稿、公开文档。
用个人模板起稿，成熟后再迁移到组织或挂到发布站点。
如果需要公开发布，可以保留单人组织，把 Site、GitHub 同步和 SEO 集中到这个组织里。

组织如何协作

必要时按业务线建 Collection，例如产品文档、帮助中心、客户交付，再把多个 Space 归入集合统一管理。
把模板、审批、GitHub 仓库、SEO 规则和 AI 工具策略都配置到组织级，再下沉到 Space。
公共 Space 负责对外发布，私有 Space 负责内部沉淀和评审，不把两种状态混在一起。

SoulDoc 的深挖点

AI 是独立用户类型，与人类共享角色和权限，不再被当成外挂助手。
Rust + SurrealDB 作为主栈，把关系、文档、图结构、向量与全文检索压进同一套底座。
审批、流程、回收站、恢复、版本、SEO 与 GitHub 同步是同一条治理链，不是散落的小功能。
模板、语言版本、发布站点、GitHub 仓库都能作为一等对象进入版本、审计和同步链。

💬 对话到文档工作流

1. 意图表达
人类或 AI 先给出原始对话、会议记录、需求片段或零散想法。
2. 工作流预览
系统先展示将生成哪些章节、标签、语言版本和待办，而不是直接静默写入。
3. 草稿生成
进入文档草稿、版本快照与变更请求链路，保持所有生成动作可审计、可回滚。
4. 治理发布
审核通过后再进入发布、SEO、多语言与公开阅读页，而不是把生成和发布混在一起。

🛡️ 治理与恢复信号

所有人类用户与 AI 用户共享同一套角色与权限模型，但动作会带上类型标识并进入审计链。
文档删除默认进入回收站，保留 30 天 恢复窗口；恢复会生成新版本，不覆盖已有历史。
多语言以 `doc_group_id` 关联，阅读端遵循默认语言与回退语言策略，搜索端允许跨语言召回。
发布前统一经过 SEO、语言完整度、公开/私有可见性和 Git Sync 状态检查。
