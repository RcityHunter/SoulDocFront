# SoulDocV5 数据库表结构设计

> 2026-04-16 收口补充：
> `collection` 是 `scope` 下的可选内容分组，不是 `space` 的强制父实体；
> `site` 不拥有内容，只通过 `site_section` 和 `site_space_binding` 发布公共 `space`；
> 产品层统一称为“GitHub 同步”，技术表名可继续使用 `git_sync_*`。

## 1. 设计原则

- `site` 是发布视图边界，对外产品文案统一称为“发布站点”
- `collection` 是内容结构边界，对外产品文案统一称为“内容集合”
- `site_section` 是站点导航分区边界，对外产品文案统一称为“站点分区”
- `space` 是权限与业务主边界
- `user` 只有两种 `user_type`：`human` / `ai`
- `document_group` 是多语言聚合对象
- `document` 是单语言知识对象
- `document_version` 与 `change_request` 共同承载治理流程
- 搜索、向量、任务、发布、GitHub 同步独立表管理，不污染主业务表

## 6. 结论

如果要同时兼顾：

- 发布站点结构
- SoulDoc 的 AI 原生工作流
- 多语言独立版本治理
- 公共 / 私有空间边界
- GitHub 同步与开放平台

数据库就不能只停留在“Space + Document”两层，而必须扩展为：

`Scope 下的可选内容集合（Collection） + Space 权限边界 + 发布站点（Site）发布视图 + 站点分区（Site Section）导航层 + Site Space Binding 发布链接 + Document Group 多语言聚合 + Document 单语言对象 + Version / Change Request 治理 + AI / Search / Publish / GitHub 同步扩展表`
