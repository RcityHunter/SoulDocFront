# SoulDocV5 AI 工具配置重构方案

> 2026-04-16 收口补充：
> AI 获取上下文时应以 `Scope -> Space -> Document` 为主链；
> `Collection` 只是可选分组；
> 对外连接统一按“GitHub 同步 / Webhook / Capability Manifest”表述。

## 1. 结论先行

SoulDoc 不应该继续把 AI 能力暴露成一串离散按钮或“助手对话框”。

最佳配置建议：

- 前台只暴露 `6` 个能力族
- 后台内部保留 `16~18` 个受控子动作
- 所有高风险写操作统一走 `预览 -> 确认 -> 写入`
- 所有高耗时动作统一走异步任务模型
- 所有外部接入统一走 `REST JSON + Async Task + Webhook + Capability Manifest`

同时要明确：

- `AI` 不是“第三类系统角色”
- `AI` 是 `user_type=ai` 的用户
- 与 `人类` 共享同一套角色与权限模型

---

## 2. 为什么不能继续用扁平工具面

原始扁平工具虽然覆盖全面，但存在几个问题：

- 前端暴露会太碎
- 用户并不关心底层调用了哪个原子动作
- `delete / merge / publish / restore` 这类动作权限边界太危险
- Skill / 外部连接器需要更稳定的协议层，而不是频繁改动的按钮集合

因此：

- 原子动作适合保留在系统内部
- 用户层看到的应该是任务与结果，而不是原子动作名字

---

## 3. 最佳配置：6 个能力族

## 3.1 Context

用于建立 AI 的工作上下文。

- `context.get`
- `context.navigate`
- `permission.inspect`

### 作用

- 获取当前 `Scope / Space / Document / Block`，以及可选的 `Collection` 绑定关系
- 获取当前用户与 AI 用户的权限边界
- 获取最近版本、最近任务、最近通知状态

---

## 3.2 Knowledge

用于浏览与理解知识空间。

- `content.list`
- `content.read`
- `search.hybrid`
- `search.related`

### 作用

- 遍历 `Collection（可选）`、Space、文档树，以及发布站点 / 站点分区 / Linked Space 结果树
- 读取文档与块内容
- 做关键词 + 向量的混合检索
- 生成引用来源、相关推荐与 FAQ 上下文

---

## 3.3 Drafting

用于生成与修改草稿，但不直接改主版本。

- `draft.create`
- `draft.patch`
- `block.transform`
- `dialogue.capture`

### 作用

- 从对话生成文档草稿
- 对草稿做 patch，而不是全量覆盖
- 做摘要、大纲、标题、FAQ、标签、翻译、审校等结构化动作

### 原则

- 默认 `patch`
- 默认先返回结构化预览
- 草稿先落临时态，不直接污染已发布版本

---

## 3.4 Governance

用于版本和审批治理。

- `version.snapshot`
- `version.compare`
- `change_request.preview`
- `approval.submit`

### 作用

- 保存快照
- 生成 Diff
- 进入变更请求
- 指派 reviewer 并等待审批结果

### 原则

- AI 不直接 `merge`
- AI 不直接 `restore` 主版本
- 恢复动作也应先进入草稿或预览态

---

## 3.5 Publishing

用于 SEO 与公开站点发布。

- `seo.inspect`
- `seo.optimize`
- `publish.preview`
- `publish.release`

### 作用

- 检查 SEO 字段完整性
- 生成 canonical、description、FAQ schema 建议
- 预览站点构建结果
- 发布到公共阅读页、私有成员页或导出目标

### 原则

- 发布前必须绑定版本
- 发布前必须权限校验
- 发布后必须生成 `release record`

---

## 3.6 Connectors

用于 Git、Webhook、外部系统与能力清单。

- `git.sync`
- `connector.invoke`
- `knowledge.retrieve`
- `webhook.emit`

### 作用

- GitHub 仓库双向同步
- 外部系统连接器调用
- 对外提供知识召回与当前页上下文
- 对外发送任务完成、发布成功、评论等事件

---

## 4. 推荐接口形式

## 4.1 同步 REST

适合：

- 读上下文
- 读文档
- 列目录
- 权限校验
- 轻量搜索

## 4.2 异步任务

适合：

- 摘要、大纲、FAQ、翻译、审校
- 批量 embedding
- 批量 SEO 优化
- 站点构建与发布

统一模式：

- `POST /ai/...`
- 返回 `task_id`
- `GET /ai/tasks/:taskId`
- 完成后触发通知 / webhook

## 4.3 Capability Manifest

建议为外部连接器提供：

- `/.well-known/capabilities.json`
- `/.well-known/skill-manifest.json`

内容包括：

- 可用能力族
- 子动作
- 输入输出格式
- 权限要求
- 是否需要确认
- 是否异步

---

## 5. 权限设计建议

## 5.1 AI 默认禁止的动作

- 直接 `delete`
- 直接 `publish`
- 直接 `merge`
- 直接覆盖主版本

## 5.2 AI 默认允许的动作

- 读上下文
- 读文档
- 搜索
- 生成建议
- 创建草稿
- 生成 Diff 预览

## 5.3 高风险动作统一策略

统一走：

`Preview -> Confirm -> Commit -> Audit Log`

---

## 6. 与 UI 的映射

## 6.1 用户层

用户不需要看到 16 个原子动作按钮。

用户应该看到的是：

- 生成摘要
- 重构大纲
- 补 FAQ
- 找相关文档
- 批量翻译
- 发起变更请求
- 发布前检查

## 6.2 系统层

系统内部仍可以保留原子动作进行编排。

例如用户点击“补 FAQ”，后端可内部串联：

- `context.get`
- `content.read`
- `search.related`
- `draft.patch`
- `change_request.preview`

---

## 7. SoulDocV5 的额外强化点

### 7.1 AI 用户平权

AI 不是助手挂件，而是与人类并列的用户类型。

### 7.2 多语言直接接入 AI 治理链

- AI 可生成翻译草稿
- 人类或具备审批权限的 AI 用户负责复审
- 语言版本状态进入通知中心与发布链

### 7.3 GitHub 同步与 AI 一起被纳入审计链

AI 改写、Git 回灌、发布构建都应沉淀成：

- 任务记录
- 版本快照
- 通知回执
- Webhook 事件

---

## 8. 最终建议

### 用户层

只展示 6 个能力族下的高频动作。

### 系统层

保留 16~18 个原子动作，用于内部编排、审计和日志追踪。

### 平台层

统一输出：

- REST API
- Async Task
- Webhook
- Capability Manifest

### 安全层

所有高风险动作都必须经过 preview 和 approval。
