# SoulDocV5 技术栈建议（Rust 预留）

## 1. 说明

本轮先把研发文档骨架整理齐，不强行锁定具体实现框架。

但如果后续进入前后端开发，建议技术栈文档提前预留，尤其是：

- 前端框架
- 后端语言与 Web 框架
- 数据库
- 搜索与向量
- 对象存储
- 队列与异步任务
- 发布与预览环境

本文件先给出建议方向。经过 `2026-04-16` 的最终收口，后端主方向已明确为 `Rust + SurrealDB`，其余组件按是否真的需要再补。

---

## 2. 前端建议方向

## 2.1 基础框架

建议优先考虑：

- `React + TypeScript`
- 方案可选：
  - `Next.js`
  - `Vite + React`

### 原则

- 不改变 `V5` 已确认的视觉与布局逻辑
- 组件化时沿用当前页面结构，而不是重新设计 IA

## 2.2 前端关键能力

- 路由与页面壳
- 状态管理
- 数据请求与缓存
- 富文本 / 块编辑器
- Diff 视图
- 任务轮询
- 多语言切换
- 公开阅读页渲染

## 2.3 前端可预留依赖

- 数据请求：`TanStack Query`
- 轻量状态：`Zustand`
- 编辑器：`TipTap` / `ProseMirror`
- 表格与列表：按业务自定义，不急着引 UI 大框架

---

## 3. 后端建议方向（Rust 预留）

## 3.1 当前建议

后端语言明确为：

- `Rust`

数据库底座明确为：

- `SurrealDB`

## 3.2 可优先评估的 Rust 组合

- Web：`axum`
- Async Runtime：`tokio`
- DB Access：`surrealdb` Rust SDK
- Auth / Session：自研中间件或成熟 crate 组合
- 配置：`serde` + `config`
- Queue / Rate Limit：`redis` 或 `NATS`

说明：

- `SurrealDB` 优先承担关系、图链接、全文索引、向量索引和事件结果写回
- 只有在规模上来后，再考虑外接专用检索组件

---

## 4. 数据与基础设施建议

## 4.1 主数据库

- `SurrealDB`

用于承载：

- 用户、组织、内容集合、Space、文档、版本、变更请求
- 多语言文档组与文档关系
- 发布站点、站点分区、Space 链接关系
- AI 任务、模板、GitHub 绑定、Webhook 配置
- 图关系、全文索引、向量召回与多维检索

## 4.2 缓存与任务

- `Redis`

用于承载：

- 会话 / 短期缓存
- 任务状态缓存
- 限流

## 4.3 对象存储

- `S3 / MinIO` 兼容对象存储

用于承载：

- 文件资产
- OG 图
- 导出产物
- 发布构建静态文件

## 4.4 搜索与向量

当前建议优先合并到 `SurrealDB`：

- 全文检索：用于标题、摘要、标签、正文片段和 SEO 元数据
- 向量检索：用于语义搜索、跨语言召回、模板推荐和 AI 上下文拼装
- 图关系查询：用于 `Scope -> (Collection?) -> Space -> Document` 与 `Site -> Section -> Linked Space` 结构穿透

只有在搜索规模和召回复杂度明显上升时，再补独立检索引擎。

---

## 5. AI 与异步任务建议

## 5.1 AI 网关

建议做统一 `AI Gateway` 层，避免业务服务直接耦合模型厂商。

职责：

- 模型选择
- 计费与额度
- Prompt 模板
- 审计日志
- 重试与超时

## 5.2 异步任务

以下能力不应走长同步请求：

- 摘要
- 大纲
- FAQ
- 翻译
- 审校
- SEO 检查
- 站点构建
- GitHub 同步

---

## 6. 预览与发布环境建议

- `Preview Environment`
  - 用于 CR / PR 预览
- `Public Site`
  - 公共空间公开结果页
- `Member Site`
  - 私有空间成员阅读页
- `Static Export`
  - HTML / PDF 导出

说明：

- 发布环境要和 `scope / collection / space + site / section / linked_space / release` 模型对齐
- 不建议把发布实现成单一按钮逻辑

---

## 7. 为什么这一轮不锁死 Rust 方案

因为目前更重要的是先把：

- 产品边界
- 服务边界
- 数据模型
- 异步任务模型
- 发布与 GitHub 同步链路

定义清楚。

只要这些边界稳定，后续换成 Rust 实现是顺的。

反过来，如果边界没定清楚，先定框架也只会反复返工。

---
