# SoulDocV5 前后端模块拆解

> 2026-04-16 收口补充：
> `Space` 是内容主容器，`Collection` 只是 Scope 下的可选管理分组；
> `Site` 只负责发布层，通过 `site_section + site_space_binding` 链接公共 `Space`；
> 产品层统一写作“GitHub 同步”。

## 1. 拆解原则

SoulDoc 不是单一编辑器，而是一个包含协作、治理、发布、公开站点、AI 与外部连接器的平台型产品。

因此拆解必须围绕以下业务域：

- Scope / Space / 内容集合（Collection，可选） / 发布站点（Site） / 站点分区（Section）
- Document / Version / Change Request
- Permission / Member / Asset
- Search / AI / Localization
- SEO / Publish / Public Site
- GitHub 同步 / Webhook / Capability Manifest
- Install / Auth / Settings

## 5. 研发约束

## 5.1 页面必须落到服务域

原型中的主页面都必须有明确服务承接，避免出现“页面有，后端域没有”的假功能。

## 5.2 AI 用户不是额外系统

`AI` 与 `人类` 共用 `user`、`role`、`space_member` 模型。

不要再为 AI 另起一套账号体系。

## 5.3 多语言必须是独立文档版本

不要把多种语言直接塞进一份正文。

后端必须围绕：

- `document_group`
- `document(locale)`
- `translation_status`
- `fallback_locale`

来设计。

## 5.4 发布链与编辑链解耦

编辑态不直接污染发布态。

发布必须基于：

- 版本快照
- 变更请求
- 或明确的发布版本

## 5.5 高风险动作统一走治理链

以下动作不能直接同步提交：

- 删除
- 合并
- 发布
- 恢复主版本

必须统一走：

`Preview -> Confirm -> Commit -> Audit`

## 6. 推荐研发顺序

### 第一阶段

- 安装、认证
- Scope / Space / Collection（可选） / Site 基础结构
- 文档树、阅读、编辑
- 版本保存
- 搜索基础能力

### 第二阶段

- 变更请求
- 成员与权限
- 标签、文件、评论
- AI 任务模型
- 语言版本模型

### 第三阶段

- SEO 与发布
- 公开阅读页
- GitHub 同步
- 开发者平台 / Webhook / Capability Manifest

### 第四阶段

- 站点级运营分析
- 更细粒度连接器
- 发布质量分析
- 多站点、多 Collection 扩展

- 明确外部连接器域
