# SoulDocV5 原型展示范围说明

> 2026-04-16 收口补充：
> `Collection` 是 `Scope（个人工作区 / 组织）` 下的可选管理分组，`Space` 才是内容主容器；
> `Site` 是发布站点，通过 `Section` 链接公共 `Space`；
> 产品可见层统一使用“GitHub 同步”，技术实现层可继续保留 `git_sync` 命名。

## 1. 目标

这一版原型不是把所有研发文档都直接可视化，而是把用户真正会接触到的产品结构、关键流程、权限边界和结果态做成可跳转页面。

原型重点：

- 保留 `V4` 已经成立的使用态页面与交互密度
- 补回 `V2` 更强的平台骨架：`Scope / Space / Document + 内容集合（Collection，可选） + 发布站点（Site） / 站点分区（Section） / Public Site / GitHub 同步 / Developer Platform`
- 明确 SoulDoc 的核心差异化：
  - 用户只有两类：`人类` 与 `AI`
  - 两类用户地位平等，共享同一套角色与权限体系
  - 空间只有两类可见性：`公共` / `私有`
  - 多语言采用“同一文档组 + 每语言独立版本”的治理模型

## 2. 展示原则

### 2.1 必须展示

- `Scope -> Space -> Document` 的内容关系、`Collection` 的可选分组角色，以及 `Site / Section / Linked Space` 的发布关系
- 空间运营视角：空间概览、成员、标签、文件、活动
- 文档主链：文档中心、编辑器、版本、变更请求
- AI 主链：AI 任务中心、AI 工具配置、搜索、语言版本
- 发布主链：SEO 与发布、公开阅读页、GitHub 同步、开放平台
- 系统层：安装、登录、系统设置、个人中心、通知中心

### 2.2 只做摘要展示

- 权限原则
- 接口分组
- 发布流程
- GitHub 同步流程
- AI 工具分层
- 开放平台能力边界

这些内容适合在页面中做“结构与入口说明”，不适合直接展开成技术表格墙。

### 2.3 不直接放进原型页

- 全量数据库表结构
- 全量接口字段与请求示例
- Repository / Middleware / Infrastructure 级实现细节
- 错误码全集
- 队列、缓存、对象存储、索引分片等底层细节

这些内容应进入单独的研发文档。

## 5. SoulDocV5 相对普通文档系统的展示重点

### 5.1 平台级边界不是概念，而是页面化结构

- `发布站点（Site）`：发布视图、品牌入口、公开导航
- `内容集合（Collection）`：Scope 下多个 Space 的可选管理分组，不是 Space 的强制父层
- `站点分区（Section）`：发布站点下组织已发布 Space 的公开导航分区
- `Space`：业务与权限边界
- `Document`：知识对象与治理对象

### 5.2 AI 不是助手挂件，而是第一公民

- UI 中不再强调“AI 助手”概念
- `AI` 与 `人类` 同为用户类型
- 共用角色与权限模型
- 自动化行为通过执行边界与审批链控制

### 5.3 多语言不是附属功能，而是治理对象

- 每种语言是一份独立文档版本
- 同一文档组跨语言关联
- 支持默认语言、回退语言、翻译状态、并行复审

### 5.4 发布不是一个按钮，而是一条链

- 版本
- 变更请求
- SEO 配置
- 发布目标
- 公开结果页
- GitHub 同步
- Webhook / 开放平台
