# SoulDocV5 发布与工作流操作文档

> 适用范围：`SoulDocV5` 原型当前版本  
> 文档目的：把 `编辑 -> 审核 -> 预览 -> 发布 -> 公开阅读 -> GitHub 同步` 这条链整理成一份可直接给产品、设计、前后端共同使用的操作说明。

## 1. 结论先行

`SoulDocV5` 的整体工作流是连贯的，但它不是“所有事情都在一个页面里完成”的模型，而是一个分阶段的治理链。

准确说法是：

- `Space` 负责内容生产、协作、权限、版本、AI、语言版本
- `发布站点（Site）` 负责对外发布结构、域名、导航、SEO 和公开阅读结果
- `发布 & SEO` 负责正式发布动作与发布治理
- `公开阅读页` 负责展示最终对外结果态
- `GitHub 同步` 负责把发布结果接入仓库、预览分支和外部系统

所以这条链是连贯的，但分散在多个页面中按阶段完成，而不是一个大而全的单页流程。

## 2. 核心对象与职责

### 2.1 Scope

`Scope` 是最上层业务范围，分为：

- `个人工作区`
- `组织`

它负责承载：

- Space
- 可选内容集合（Collection）
- 发布站点（Site）
- 成员与角色
- 模板、审批、AI 配置、GitHub 同步配置

### 2.2 Space

`Space` 是内容主容器，也是核心工作区。

它负责：

- 承载文档与目录树
- 承载成员、角色、权限
- 承载 AI 用户与人类用户协作
- 承载标签、文件、搜索边界
- 承载版本、Change Request、审批链
- 承载多语言文档版本

一句话：`Space` 决定“谁在这里生产内容、管理内容、审核内容”。

### 2.3 Collection

`Collection` 是 `Scope` 下的可选内容分组，用来管理多个 `Space`。

它不是强制父层，也不是发布本体。

它负责：

- 在后台管理层面把多个 Space 分组
- 让大型组织的内容结构更清晰
- 作为内容治理与权限编排的辅助对象

### 2.4 发布站点（Site）

`发布站点` 是发布壳，不直接拥有文档内容。

它负责：

- 域名
- 品牌与站点名
- SEO
- 公开导航
- 公开访问策略
- 对外结果页

一句话：`Site` 决定“内容最终如何被外部阅读者看到”。

### 2.5 站点分区（Section）

`Section` 属于 `Site`，是发布站点内的导航分区。

它负责：

- 组织公开导航
- 把一个或多个公共 Space 挂接到对外站点结构中
- 形成“产品文档 / 帮助中心 / 开放平台 / 更新日志”这类栏目层

### 2.6 Document / Version / Change Request

它们都属于 `Space` 内部治理链：

- `Document`：具体文档对象
- `Version`：文档版本快照
- `Change Request`：变更请求与审批对象

## 3. 页面职责分工

当前原型中，这条链按页面分工如下：

- [spaces.html](</C:/Users/DELL/Desktop/SoulDocV5/spaces.html>)：查看个人或组织下有哪些 Space
- [space.html](</C:/Users/DELL/Desktop/SoulDocV5/space.html>)：查看某个 Space 的概览、权限、成员、活动
- [docs.html](</C:/Users/DELL/Desktop/SoulDocV5/docs.html>)：管理文档列表、筛选、创建、语言版本入口
- [editor.html](</C:/Users/DELL/Desktop/SoulDocV5/editor.html>)：编辑正文、目录树、块内容、语言与 AI 辅助
- [versions.html](</C:/Users/DELL/Desktop/SoulDocV5/versions.html>)：查看历史版本、Diff、回退
- [change-request.html](</C:/Users/DELL/Desktop/SoulDocV5/change-request.html>)：走变更请求与审批
- [workspace.html](</C:/Users/DELL/Desktop/SoulDocV5/workspace.html>)：配置发布站点、站点分区、链接公共 Space
- [seo.html](</C:/Users/DELL/Desktop/SoulDocV5/seo.html>)：执行发布、管理 SEO、查看发布历史与发布目标
- [public-doc.html](</C:/Users/DELL/Desktop/SoulDocV5/public-doc.html>)：查看最终公开阅读结果
- [git-sync.html](</C:/Users/DELL/Desktop/SoulDocV5/git-sync.html>)：查看 GitHub 仓库绑定、预览分支、同步回执

因此：

- `workspace.html` 是发布结构配置页
- `seo.html` 是正式发布执行页
- `public-doc.html` 是发布结果页

这三个页面一起才构成完整的发布链。

## 4. 端到端完整工作流

## 4.1 阶段一：确定归属范围

用户先确定这批内容属于哪里：

- 个人工作区
- 某个组织

这一步决定：

- 成员体系
- 权限边界
- 后续是否协作
- 发布站点归属

## 4.2 阶段二：创建或选择 Space

在 `Scope` 下创建或进入一个 `Space`。

这一步决定：

- 这批文档的主题边界
- 谁可以进入和编辑
- 是否允许公开发布
- AI 用户是否参与

典型例子：

- 产品文档 Space
- 帮助中心 Space
- API 参考 Space
- 运营内容 Space

## 4.3 阶段三：创建文档与目录树

进入 [docs.html](</C:/Users/DELL/Desktop/SoulDocV5/docs.html>) 与 [editor.html](</C:/Users/DELL/Desktop/SoulDocV5/editor.html>)：

- 新建文档
- 组织目录树
- 编辑正文与块内容
- 补标签、文件、引用来源
- 处理多语言版本
- 使用 AI 生成摘要、大纲、标签、FAQ 等

这一步的结果是：`Space` 内形成可治理的文档集合。

## 4.4 阶段四：版本与变更请求

文档成稿后，不直接发布，而是进入治理链：

- 生成版本
- 查看版本 Diff
- 发起 Change Request
- 指定审批人
- 处理审批意见

相关页面：

- [versions.html](</C:/Users/DELL/Desktop/SoulDocV5/versions.html>)
- [change-request.html](</C:/Users/DELL/Desktop/SoulDocV5/change-request.html>)

这一步的目的是让“可发布内容”与“正在编辑内容”分离。

## 4.5 阶段五：确认 Space 的公开属性

只有 `公共 Space` 才能被挂接到 `发布站点`。

这里有一条明确规则：

- `私有 Space`：只能在内部使用，不能直接进入公开阅读页
- `公共 Space`：可以被链接到 `Site / Section`，进入对外发布链

因此，`Space` 层首先决定“能不能被发布”。

## 4.6 阶段六：配置发布站点结构

进入 [workspace.html](</C:/Users/DELL/Desktop/SoulDocV5/workspace.html>)：

- 选择或新建 `发布站点`
- 配置域名、品牌、访问策略
- 配置 `Section`
- 把公共 `Space` 链接到指定 `Section`

这一步不是“发布动作”，而是“发布编排动作”。

用户在这里完成的是：

- 哪些内容要被发布
- 这些内容在对外站点上如何组织
- 外部读者从什么结构进入

## 4.7 阶段七：进入发布 & SEO

进入 [seo.html](</C:/Users/DELL/Desktop/SoulDocV5/seo.html>)：

- 选择发布目标
- 配置站点 SEO
- 配置单文档 SEO
- 查看待发布文档
- 触发单篇发布、批量发布或全量发布
- 查看发布历史

这里是当前原型里真正执行“发布”的页面。

也就是说：

- `workspace.html` 不负责点击“确认发布”
- `seo.html` 才负责点击“发布”

## 4.8 阶段八：预览与公开结果确认

发布后，进入 [public-doc.html](</C:/Users/DELL/Desktop/SoulDocV5/public-doc.html>)：

- 检查对外路径是否正确
- 检查公开导航是否正确
- 检查多语言切换是否正确
- 检查公共 / 私有访问结果是否正确
- 检查 SEO 预览结果是否符合预期

这一页是最终结果态，而不是管理态。

## 4.9 阶段九：GitHub 同步与外部分发

如果启用了 docs-as-code 或外部分发链，就继续进入 [git-sync.html](</C:/Users/DELL/Desktop/SoulDocV5/git-sync.html>)：

- 绑定 GitHub 仓库
- 关联分支与预览环境
- 查看同步结果
- 处理冲突与回灌
- 对接 Webhook 或外部系统

这一步把发布结果接入工程化链路。

## 5. 这条工作流是否连贯

是连贯的，而且应该被理解为“一条分阶段、逐步推进的治理链”。

它不是：

- 编辑完直接公开
- 一个页面做完所有事情
- 内容层和发布层混在一起

它是：

1. 在 `Space` 内完成内容生产
2. 在 `Version / Change Request` 内完成治理
3. 在 `发布站点` 内完成编排
4. 在 `发布 & SEO` 内完成正式发布
5. 在 `公开阅读页` 内确认结果
6. 在 `GitHub 同步` 内完成外部分发

所以答案是：`连贯，但分阶段；一步一步进行，但每一步由不同页面承载。`

## 6. Space 层工作流与状态展示

`Space` 是内容与协作主边界，因此它有一套自己的状态。

### 6.1 Space 的核心状态

- `可见性`：公共 / 私有
- `归属范围`：个人工作区 / 组织
- `协作状态`：单人 / 多成员 / AI 协作中
- `内容状态`：草稿中 / 审核中 / 可发布 / 已部分发布 / 发布阻塞
- `发布关系`：未挂接 / 已挂接到发布站点

### 6.2 Space 状态在原型里的展示位置

- [spaces.html](</C:/Users/DELL/Desktop/SoulDocV5/spaces.html>)：展示 Space 列表与入口
- [space.html](</C:/Users/DELL/Desktop/SoulDocV5/space.html>)：展示成员、权限、标签、活动与概览
- [members.html](</C:/Users/DELL/Desktop/SoulDocV5/members.html>)：展示人类用户 / AI 用户与角色关系
- [docs.html](</C:/Users/DELL/Desktop/SoulDocV5/docs.html>)：展示文档数量、语言、发布准备度
- [change-request.html](</C:/Users/DELL/Desktop/SoulDocV5/change-request.html>)：展示审核链状态

### 6.3 Space 层关键规则

- Space 可以独立存在，不必立即发布
- 私有 Space 默认不能进入公开发布链
- 公共 Space 才能被挂接到 `Site / Section`
- 文档、版本、审批、AI 都属于 Space 层，不属于 Site 层

## 7. 发布站点层工作流与状态展示

`Site` 是发布壳，因此它关注的是“站点配置”和“发布结果”，不是“编辑正文”。

### 7.1 发布站点的核心状态

- `站点配置状态`：未配置 / 已配置
- `分区状态`：未建分区 / 已建分区 / 已绑定公共 Space
- `SEO 状态`：未补齐 / 已补齐
- `发布状态`：未发布 / 预览就绪 / 发布中 / 已上线 / 发布失败 / 已回滚
- `同步状态`：未接 GitHub / 已接入 / 冲突中 / 已同步

### 7.2 发布站点状态在原型里的展示位置

- [workspace.html](</C:/Users/DELL/Desktop/SoulDocV5/workspace.html>)：展示 Site、Section、Linked Space 关系
- [seo.html](</C:/Users/DELL/Desktop/SoulDocV5/seo.html>)：展示发布目标、待发布文档、发布历史与 SEO 完整度
- [public-doc.html](</C:/Users/DELL/Desktop/SoulDocV5/public-doc.html>)：展示最终公开结果
- [git-sync.html](</C:/Users/DELL/Desktop/SoulDocV5/git-sync.html>)：展示同步与分发状态

### 7.3 发布站点层关键规则

- Site 不直接拥有 Document
- Site 只能通过 `Section` 去链接公共 `Space`
- Site 解决的是“怎么对外展示”，不是“谁来写内容”
- 站点级发布历史与回滚应独立可追踪

## 8. 当前原型里的发布按钮到底在哪里

用户会觉得“发布站点页像缺一个发布按钮”，这个感受是对的，但要区分：

- 从系统功能上说：`不缺`
- 从单页感知上说：`workspace.html` 确实不像最终操作页

当前原型里的发布按钮在 [seo.html](</C:/Users/DELL/Desktop/SoulDocV5/seo.html>)：

- `全量发布`
- `批量发布`
- 单目标 `发布`
- 单文档 `发布`

因此当前设计是：

- [workspace.html](</C:/Users/DELL/Desktop/SoulDocV5/workspace.html>)：配置发布结构
- [seo.html](</C:/Users/DELL/Desktop/SoulDocV5/seo.html>)：正式执行发布

这是功能层面的完整链，但从体验上，`workspace.html` 最好继续保留明显的：

- `进入发布`
- `查看公开阅读页`
- `GitHub 同步`

让用户知道下一步去哪。

## 9. 建议采用的标准操作顺序

推荐对外统一培训和产品说明时使用以下顺序：

1. 先选 `个人工作区` 或 `组织`
2. 再进入或创建 `Space`
3. 在 `文档中心 / 编辑器` 里写文档
4. 通过 `版本 / Change Request` 做审核
5. 确认 `Space` 是否为公共
6. 在 `发布站点` 页面挂接站点分区
7. 在 `发布 & SEO` 页面做发布前检查与正式发布
8. 在 `公开阅读页` 查看最终结果
9. 如有需要，再走 `GitHub 同步`

这是当前 `SoulDocV5` 最清晰、最稳定、最容易培训团队理解的主流程。

## 10. 单人使用与组织使用的区别

### 10.1 单人使用

单人场景下，这条链会更轻：

- 在个人工作区创建一个或多个 Space
- 自己编辑、自己审核、自己发布
- 可选择是否建立发布站点
- 可选择是否接 GitHub

### 10.2 组织使用

组织场景下，这条链会更重：

- 多成员协作
- AI 用户作为独立账户参与
- 更明确的审批、权限、角色
- 多个公共 Space 被统一挂到一个发布站点
- 更需要模板、SEO、GitHub 同步和公开治理

## 11. 对研发的实现含义

前后端开发时，必须把这几层拆开实现：

- 内容层：`Space / Document / Version / Change Request / Locale`
- 管理层：`Scope / Collection / Member / Role / Permission`
- 发布层：`Site / Section / SiteSpaceBinding / Release / SEO`
- 分发层：`GitHub Sync / Webhook / Preview Environment`

不要把这些对象混在一个“大文档表”或“单页状态机”里，否则后续一定会失控。

## 12. 最终结论

`SoulDocV5` 当前的工作流是成立的，主链可以稳定定义为：

`Scope -> Space -> Document -> Version / CR -> Site / Section -> Publish & SEO -> Public Reader -> GitHub Sync`

其中：

- `Space` 是内容生产与治理主边界
- `Site` 是发布与公开阅读主边界
- `SEO` 页是发布动作主入口
- `公开阅读页` 是最终结果态

后续如果继续优化原型，重点不是重做结构，而是继续强化页面之间的“下一步动作提示”，让用户更自然地理解自己正处在这条链的哪一步。
